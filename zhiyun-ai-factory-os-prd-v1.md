## 0. 阅读指南：本PRD与V3演示原型的关键差异

进入具体需求前，先明确几个原则性调整，避免团队按演示原型的表面效果理解需求：

| 维度 | V3演示原型 | 本PRD |
|---|---|---|
| AI Copilot | 关键词匹配的固定话术 | P1做规则化意图识别+结构化查询；P3引入大模型+企业知识库RAG，且要求"回答必须可追溯到具体数据" |
| 生产排产 | 写死的甘特图数据 | P1实现约束规则+启发式排产算法，产出"可行解"而非"最优解"，AI优化作为人工可采纳的辅助建议 |
| 根因分析/利润归因 | 固定的百分比归因结果 | 先做"关联数据呈现"，最终判断交给人工；置信度必须标注计算依据 |
| "AI自我进化"+跨厂知识蒸馏 | 展示模型微调曲线和联邦学习式描述 | 本期不做，替换为务实的"决策日志+采纳率+效果回溯"看板；联邦学习级能力列为远期方向 |
| 模块范围 | 16个模块同时展示 | 拆分为P0-P3四期交付，每期必须是自身可验证的完整闭环，不是功能拼盘 |
| 数据来源 | 无采集入口，数据预置 | 新增"车间数据采集层"（M0）作为P0地基模块 |
| 安全资质 | 登录页展示"等保三级" | 移出产品需求范围，作为独立合规里程碑单独跟踪，测评通过前不得对外宣称 |

## 1. 产品概述

### 1.1 背景与问题
中小型服装/制衣代工厂普遍面临：订单交期敏感、多客户多SKU并行、车间数据依赖人工汇报或事后补录、异常发现滞后（往往客户投诉后才发现问题）、老板/计划员凭经验决策难以复用。本产品的目标不是把工厂管理"AI化"作为噱头，而是缩短"发现问题"到"处理完成"的时间，并让处理过程系统性地被记录下来，形成可复用的经验资产。

### 1.2 产品定位
面向中小型服装代工厂的经营管理与生产决策辅助系统。核心方法论：**先把数据流跑通，再谈自动化；先做"辅助建议"，再谈"自主执行"**。

### 1.3 目标客户画像
- 规模：50-500名生产工人，年产值2000万-2亿元人民币
- 业务模式：多客户贴牌/来料加工，小批量多批次或中批量定制混合
- 现状：以Excel、微信群、纸质流转单为主，可能有基础财务软件，生产端基本无数字化系统
- 决策者：厂长/老板通常也是最终拍板人，对"立竿见影"的效果比对"功能全面"更敏感

### 1.4 用户角色与核心诉求

| 角色 | 核心诉求 | 主要使用模块 |
|---|---|---|
| 超级管理员/厂长 | 全局经营视图、异常预警、决策效率 | 数据看板、预警中心、经营分析 |
| 计划员 | 排产合理、交期可控 | 生产排产、订单中心 |
| 采购/跟单 | 物料齐套、供应商稳定 | 进销存、采购管理 |
| 仓管 | 出入库准确、库存可追溯 | 进销存 |
| 生产/裁剪主管 | 工序效率可见、异常及时处理 | 生产执行监控、车间数据采集端 |
| 质检 | 检验记录规范、问题可追溯 | 品控中心 |
| 财务 | 成本核算准确、结算及时 | 财务结算 |

### 1.5 产品目标
- 业务目标（示例，需在POC阶段结合真实客户校准基线）：缩短"异常发生"到"人工介入处理"的平均时间；提升订单交期达成率；降低因排产/物料信息滞后导致的产能空转或紧急外协成本
- 非目标：不承诺"替代计划员/采购/质检的判断"，AI角色被明确限定为"辅助建议"（见第2章分级说明）

### 1.6 本期范围与非目标
**本期（P0-P2）范围**：接单→排产→生产→质检→入库→发货→结算主链路的数字化与半自动化辅助，覆盖单工厂场景。

**明确不做（Out of Scope）**：多工厂/跨工厂协同调度；面向消费者/品牌方的分销网络与C端功能；完全自主决策（L4）的AI执行；联邦学习/跨企业模型协同；计算机视觉自动质检（列为P3+方向，不在本次范围）。

## 2. 核心设计原则：如何保证"闭环"

这是本PRD相对原型最重要的部分。原型的问题不是功能不够多，而是很多功能只做到了"展示结果"，没有说明结果从哪来、执行完之后又流向哪去。因此，**任何功能点在设计和评审时必须显式回答以下四个问题，否则不能视为需求完整**。

### 2.1 闭环四要素

| 要素 | 说明 | 原型的典型缺失例子 |
|---|---|---|
| ① 输入来源 | 数据从哪来：人工录入/设备或扫码采集/其他模块流转 | "生产执行监控"展示了工序效率数字，未说明数字如何产生 |
| ② 处理逻辑 | 用规则引擎/统计方法/机器学习/大模型中的哪一种，AI介入到什么等级 | "AI排产"直接展示排产结果，无算法或规则说明 |
| ③ 输出动作 | 呈现给谁、什么形式、触发什么后续 | 多数模块做到了这一步（原型UI层面做得最好的部分） |
| ④ 结果回流 | 执行后的真实结果如何被记录，并反过来修正②的逻辑或参数 | 几乎所有"AI建议"模块都缺失，是原型最大的系统性问题 |

**评审规则**：任何需求若无法说明④的具体机制，一律先按 L0/L1（见下）设计，不得包装成"智能预测/智能优化"对外呈现。

### 2.2 AI介入分级（L0-L4）

为避免"数字员工替代人"的叙事在需求阶段被无限放大，所有涉及"AI"字样的功能必须先标注等级：

- **L0 数字化记录**：只做记录、检索、展示，无判断逻辑
- **L1 规则提示**：基于预设阈值/规则触发提醒（如库存低于安全线），逻辑可解释、可配置
- **L2 建议+人工确认（本期默认等级）**：系统生成建议方案并附依据说明，人工"采纳/修改/拒绝"后才真正执行
- **L3 自动执行+事后可撤销**：仅限影响范围小、规则完全确定的场景（如生成待办、发送催货短信），必须有完整操作日志和撤销入口
- **L4 完全自主决策**：本期不做，任何模块都不允许出现无需人工确认的资金/排产/物料变更类自动执行

### 2.3 需求评审闭环Checklist

每个功能模块进入开发前，产品经理需完成自查：
1. 输入数据现在真实存在吗？如果不存在，采集能力是否已在M0规划中？
2. 处理逻辑属于L0-L4中的哪一级？为什么不是更高或更低？
3. 谁是最终确认人？确认后系统具体执行什么动作？
4. 执行结果预计多久后可验证？验证结果存放在哪张表、由谁回填或自动采集？
5. 若验证结果与预期不符，系统/团队如何据此调整规则或参数？

## 3. 总体业务闭环（端到端主链路）

```
订单获取 → 工艺匹配(检索DNA库) → 报价核算 → 订单确认
   → 生产排产(规则+建议) → 物料齐套检查
       ├─[不齐套]→ 触发采购/外协建议 → 采购执行 → 齐套复核 ─┐
       └─[齐套]────────────────────────────────────────┤
   → 生产执行(裁剪→缝制→后整→包装，逐工序报工) → 过程数据采集(M0)
   → 品控检验
       ├─[不合格]→ 异常分析 → 整改措施 → 复检 ─┐
       └─[合格]────────────────────────────┤
   → 成品入库 → 出库发货 → 财务结算 → 客户履约反馈
   → 数据沉淀：
       · 实际用料/工时/返工率 → 回写DNA库(M2)，修正下次报价与排产参数
       · 交期达成情况 → 回写客户档案，用于未来排产优先级判断
       · 每条AI建议的采纳与效果 → 写入决策记录(M13)，作为参数迭代依据
```

**闭环判定标准**：上图中"数据沉淀"指回上游的箭头，都必须有真实数据表和写入逻辑支撑，不能只是流程图上的装饰——这是评审时的强制检查点。

## 5. 功能模块详细设计

统一格式：**功能描述 / 输入来源 / 处理逻辑与AI分级 / 输出 / 闭环反馈点**

### P0 基础设施期（先打地基，否则后续所有AI功能都是空中楼阁）

#### M0 车间数据采集层【原型缺失，本PRD新增，优先级最高】
- **功能描述**：以最低操作成本，把工序进度、物料出入库、基础质检结果从车间现场变成结构化数据
- **输入来源**：一线工人/仓管员/质检员的现场操作（移动端或车间平板扫码、计件数字键盘、语音数字输入）
- **处理逻辑**：L0，仅做结构化存储与基础校验（如单次报工数量不能超过工单剩余量）
- **输出**：实时工序进度表、物料流水账、质检原始记录
- **闭环反馈点**：本模块自身闭环体现在"录入准确率监控"——定期抽样比对报工数据与实际盘点/交付数据的偏差，偏差率是衡量地基是否可靠的核心指标
- **验收标准**：单次报工操作≤10秒；试点车间连续两周数据录入完整率≥90%

### P1 核心业务闭环期（MVP，目标：跑通接单到发货的完整闭环）

#### M1 订单管理
- **功能描述**：订单录入、状态跟踪、订单看板；对高频客户提供字段自动填充建议（历史尺码比例、常用工艺）
- **输入来源**：业务员人工录入为主；P2可接入半结构化Excel/表单导入
- **处理逻辑**：L1，基于客户历史订单数据做字段建议，不做无依据的自动生成
- **输出**：结构化订单记录，触发M3排产
- **闭环反馈点**：订单实际交付结果（准时/延期天数/是否投诉）回写订单记录与客户档案，作为后续该客户排产优先级、报价折让空间的判断依据

#### M2 工艺库（款式DNA库）
- **功能描述**：结构化保存每个历史款式的工艺路线、标准用料、标准工时、历史风险点、返工原因分布，由工艺员/质检在订单结案后维护，而非"AI自动生成"
- **输入来源**：结案订单的实际生产数据（来自M0/M4/M6）+ 工艺员人工补充经验字段
- **处理逻辑**：L1，新订单可基于款式特征做相似款检索（简单特征匹配即可，不依赖复杂模型）
- **输出**：被M1报价、M3排产、M6品控直接引用的标准参数
- **闭环反馈点**：每次生产结束后，实际用料、工时、返工率自动/半自动更新对应DNA记录——这是对"AI自我进化"最诚实也最容易落地的版本：不是模型自己变聪明，是"经验库随生产越用越准"

#### M3 生产排产
- **功能描述**：基于订单交期、在制产能、物料齐套情况生成排产方案
- **处理逻辑**：L2，第一版采用规则+启发式算法（如"最早交期优先+产能负载均衡"），产出"一个可行方案"，不对外宣称"最优解"；复杂多目标优化留待P3根据真实数据评估是否投入运筹优化专项
- **输出**：甘特图视图、待确认调整建议清单
- **闭环反馈点**：排产计划vs实际执行进度（来自M0/M4）的偏差被持续记录，用于修正各工序/班组的产能估算参数

#### M4 生产执行监控
- **功能描述**：车间工序进度、效率、异常的实时呈现
- **输入来源**：M0采集数据
- **处理逻辑**：L1，基于标准工时（来自M2）计算效率偏差并触发提醒
- **输出**：车间看板、异常提醒推送给对应主管
- **闭环反馈点**：异常处理结果（是否解决、解决耗时）记录进预警中心的效果统计

#### M5 进销存管理
- **功能描述**：物料库存、出入库记录、短缺预警
- **处理逻辑**：短缺预警L1（低于安全库存触发），采购建议L2（生成草稿，人工确认后转采购单）
- **闭环反馈点**：采购单实际到货时间vs承诺交期的偏差，回写供应商档案，形成真实的供应商准时率统计（而非展示性固定百分比）

#### M6 品控中心
- **功能描述**：检验记录、异常统计、多因素关联呈现
- **处理逻辑**：L1-L2，异常检测基于统计阈值（如某班组返工率环比上升超过设定百分点触发提醒）；原因分析呈现"关联数据"（如同期是否有新员工排班、是否更换面料批次），最终判断交给质检/主管人工完成，不做"确定性因果链+精确置信度"的展示
- **闭环反馈点**：整改措施执行后，对应工序返工率是否真的下降会被持续追踪，逐步积累"哪类整改对哪类问题真正有效"的经验记录

**P1阶段整体验收标准**：任意订单应能完整走完"录入→排产→生产→质检→入库→出库"全流程，且每一步都有系统内可追溯记录，任何角色不需在系统外（导出Excel、微信群沟通）二次留存关键信息才能完成协作。

### P2 供应链与经营协同期

#### M7 采购与外协管理
- **功能描述**：供应商档案管理、采购单/外协单全流程跟踪，与M5共享物料需求触发逻辑
- **输入来源**：M5触发的采购建议、外协车间通过M0采集端回传的进度与完工数据
- **处理逻辑**：L2，采购/外协单生成后人工确认执行；供应商推荐排序基于历史准时率/质量评分（来自真实交易记录）
- **输出**：采购单、外协单、供应商履约看板
- **闭环反馈点**：每单实际交期/质量与承诺值的偏差持续写回供应商档案，驱动下次推荐排序变化

#### M8 财务结算
- **功能描述**：订单成本核算、毛利统计、结算单生成
- **输入来源**：M1订单价格、M2标准成本参考、M5/M7实际采购成本、M0实际工时记录——强制要求成本核算直接引用以上模块数据，不允许财务另建一套手工Excel成本表
- **处理逻辑**：L1，按公式自动汇总，异常项（实际成本大幅偏离标准成本）触发提醒
- **输出**：订单级毛利报表、结算单
- **闭环反馈点**：毛利归因若需展开到具体因素，必须能逐项拆解到上述数据来源，不能像原型一样直接展示无法溯源的百分比

#### M9 审批中心
- **功能描述**：统一各模块产生的审批事项（采购单、外协单、领料单、临时排产调整等）
- **处理逻辑**：L2，人工审批，审批规则（金额阈值、多级审批）可配置
- **闭环反馈点**：审批时效与审批后实际执行结果的关联记录，用于识别流程瓶颈

### P3 智能化增强期（前提：P0-P2已积累一定周期的真实运行数据）

#### M11 AI Copilot（真正的自然语言交互）
- **功能描述**：接入大模型，对M1-M10的结构化数据做检索增强生成（RAG）
- **处理逻辑**：L1-L2，强制要求"每条回答标注数据来源"，不允许模型在无数据支撑时编造经营结论；涉及执行类请求（如"生成采购单"）一律进入L2人工确认，不做自动下单
- **闭环反馈点**：用户对AI回答的反馈（有用/无用/纠错）被记录，用于优化检索策略和提示词

#### M12 预警中心
- **功能描述**：汇总M3-M6各模块触发的规则提醒，按优先级排序展示
- **闭环反馈点**：预警处理时效、预警准确性（有无误报/漏报）的持续统计

#### M13 决策记录与效果追踪看板【"AI自我进化"叙事的诚实落地版】
- **功能描述**：记录系统给出的每条L1及以上建议：建议内容、依据、人工处理方式、执行后真实结果、预期与实际的偏差
- **闭环反馈点**：本模块即全系统的"闭环仪表盘"——按月/季度生成各类建议的真实采纳率和效果达成率，作为是否值得投入更高等级AI能力的决策依据。**不承诺"周准确率自动提升"，只承诺"每条建议的效果都可查、可复盘"**
- **验收标准**：任意一条历史AI建议，都能在系统中查到"后来发生了什么"

#### M14 组织权限管理
- 支持角色自定义、细粒度权限配置，闭环点在于权限变更操作日志的完整留存

## 6. 关键闭环流程详解

### 6.1 交期风险预警闭环

| 步骤 | 说明 |
|---|---|
| 1. 感知 | 系统每日基于M0实际进度与M3排产计划自动比对差异 |
| 2. 触发 | 差异超过预设阈值（如滞后超过该工序标准工时20%），标记为风险订单 |
| 3. 生成建议 | 从预设应对策略库匹配可选方案（调整排产顺序/申请外协/主动联系客户协调交期），展示每个方案的预计影响 |
| 4. 人工确认 | 计划员选择方案或自定义调整，可记录选择理由 |
| 5. 执行 | 系统更新排产计划/生成外协申请/创建客户沟通待办 |
| 6. 结果回流 | 订单最终是否延期、预警是否命中被记录，用于季度复盘预警阈值是否合理（太松漏报、太紧则告警疲劳） |

### 6.2 品质异常处理闭环

感知（M0质检数据异常波动）→ 触发（统计阈值提醒）→ 关联信息呈现（人员排班/物料批次/设备参数变化）→ 人工判断根因 → 制定整改措施 → 执行 → 复检验证效果 → 有效则记录进M13供未来参考，无效则进入下一轮排查。

### 6.3 物料短缺采购闭环

库存监控（M5）→ 低于安全线触发预警（L1）→ 生成采购建议草稿，标注预计到货时间与备选供应商（L2）→ 人工确认下单 → 到货验收，实际到货时间/质量回写供应商档案 → 供应商准时率/质量评分更新 → 影响下次同类物料的供应商推荐排序。

## 7. 核心数据实体与关键字段（体现闭环设计）

以下仅列出与"闭环"直接相关的关键字段，完整数据字典由技术团队在详细设计阶段补充。

**订单表（Order）**：除基础字段外，闭环相关字段为**预计交期/实际交期**、**预计成本/实际成本**——毛利归因的前提是这两组数字都有真实来源，不能一个系统算、一个手工填。

**AI决策记录表（Decision Log，M13核心表）**

| 字段 | 说明 |
|---|---|
| 决策ID | 唯一标识 |
| 触发模块/规则 | 说明是哪个规则/模型触发，禁止"AI综合判断"这类不可解释描述 |
| 建议内容 | 具体建议文本 |
| 依据数据 | 引用了哪些数据/规则，需可追溯 |
| 置信度 | 若展示置信度，须注明计算方式（如基于历史同类建议的达成率），不能是展示性数字 |
| 处理人/处理方式 | 采纳/修改/拒绝 |
| 执行时间 | - |
| 预期效果 | 量化描述 |
| 实际效果（回填时间点T+N） | 需设定明确验证时间点，由系统核对或人工回填 |
| 偏差说明 | 若实际与预期不符，记录可能原因 |

**款式DNA记录表（Style DNA）**：标准工时/标准用料需同时保留"当前值"与"历史更新记录"，确保能看到该数字如何随实际生产数据修正——这是"越用越准"闭环的直接证据。

**供应商档案表（Supplier）**：准时率、质量评分字段必须有原始交易记录支撑，不能是初始录入后不再更新的静态评分。

## 8. 非功能需求

- **性能**：车间移动端操作（扫码报工等）响应时间应≤1秒，避免因等待造成一线人员放弃使用
- **数据安全与权限**：分角色、分数据范围的访问控制；关键操作（审批、AI建议采纳、权限变更）需留存不可篡改的操作日志
- **合规**：如涉及信息安全等级保护等资质要求，作为独立项目里程碑由专门团队走真实测评/备案流程，测评通过前任何对外材料不得宣称已具备该资质
- **部署方式**：优先SaaS云端交付以控制客户侧IT成本；对数据隐私敏感的外贸/出口客户，评估是否需要私有化部署选项
- **可用性**：车间网络环境可能不稳定，M0采集端需支持弱网/离线缓存后自动同步

## 10. 成功指标与验收标准

不沿用原型中未经验证的固定百分比（如91%准确率），而是建立"先设定基线、再谈改进目标"的指标框架：

| 指标类别 | 示例指标 | 说明 |
|---|---|---|
| 数据基础 | M0录入完整率、录入偏差率 | P0阶段核心验收依据 |
| 流程闭环 | 订单全流程系统内可追溯比例 | 目标：P1上线后达到100%（不再需要系统外二次记录） |
| 决策质量 | AI建议采纳率、采纳后效果达成率 | 需运行至少一个季度建立基线，再设定改进目标 |
| 业务结果 | 交期达成率变化、异常处理时效变化 | 与试点工厂共同定义变化前基线 |

## 11. 风险登记表

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 车间一线人员对新增录入操作有抵触 | M0数据完整率低，全链路AI能力失去数据基础 | 从操作成本最低的场景切入（如仅扫码不需打字）；与计件工资等现有激励机制结合，而非单纯增加考核 |
| 目标客户数据基础薄弱（历史数据缺失/混乱） | 款式DNA库、成本核算等模块初期数据质量差 | P0/P1阶段预留数据清洗与老师傅经验录入的实施服务，不能假设客户能自行完成数据准备 |
| 赛道已有同类竞品（面向服装行业的现有ERP/MES厂商） | 差异化不清晰导致获客困难 | 正式立项前完成竞品拆解，明确不是"功能对齐"而是"闭环颗粒度和实施成功率"上的差异化 |
| 团队高估AI能力成熟度，按原型的"自动化"叙事排期 | 项目延期或交付效果与承诺严重不符 | 严格执行第2章AI分级评审机制，任何"智能XX"功能上线前必须过闭环Checklist |
| 范围蔓延，试图一次性做全16个模块 | 资源分散，无一模块能打磨到位 | 严格按P0-P3阶段交付，每期结束需有真实工厂验证结果才能进入下一期 |
